# Website Visual Test v1.0.0

Release date: 2026-07-01

## Overview

Website Visual Test v1.0.0 is the first stable release of the local Playwright-powered visual comparison tool. It is ready for repeatable reviewer workflows where a non-technical user needs to compare the same relative URLs across two website domains, inspect visual differences, and share evidence without using the command line.

This release keeps the repository generic and client-neutral. Runtime presets, generated screenshots, reports, and URL inventories remain outside tracked source files so comparison work can be run for different projects without committing private or client-specific URLs.

## What Changed Since v0.1.0

### More Stable Screenshots

- Automatically attempts to accept or dismiss common cookie consent banners before capturing screenshots.
- Includes default selectors for common OneTrust-style consent controls and other generic consent buttons.
- Waits for page images to complete loading before screenshot capture.
- Adds image readiness timeout controls so slow assets do not hang a run indefinitely.
- Keeps screenshot stabilization configurable through `visual.config.ts`, environment variables, and CLI options.

### Clearer Result Evidence

- Records per-comparison duration in run metadata.
- Shows the time taken for each page and viewport in live dashboard rows.
- Shows per-comparison duration in generated HTML and Markdown reports.
- Captures and displays final post-navigation baseline and target URLs, so redirects are visible in the result instead of hidden behind the requested URL.
- Falls back to the requested URL for older reports that were generated before final-URL metadata existed.

### Faster Review Triage

- Adds status filters to live dashboard results:
  - `All`
  - `Failed`
  - `Success`
  - `Errors`
  - `Skipped`
- Adds the same status filters to generated HTML reports.
- Adds search across page path, viewport, requested URLs, final URLs, status, and notes.
- Keeps active live-result filters applied while a job continues polling for new results.
- Shows visible result counts while filtering.

### Report Experience

- HTML reports include zoomable screenshot previews for baseline, target, and diff images.
- Reports can be opened from the dashboard, from historical runs, or from exported ZIP bundles.
- Static HTML reports remain self-contained enough for filtering and lightbox review without needing the local server, as long as the screenshot files are present beside the report.
- Markdown summaries include the key result evidence for sharing or issue notes.

### Dashboard Workflow

- Local dashboard runs at `http://127.0.0.1:4317/` by default.
- Non-technical users can:
  - enter baseline and target domains,
  - select sitemap, crawl, both, or manual URL discovery,
  - paste relative URL lists,
  - save and reload local presets,
  - run URL extraction,
  - run visual comparison,
  - watch progress and logs,
  - inspect live results before the full job completes,
  - review previous runs,
  - download report ZIP bundles.

## Supported Workflows

### Dashboard Workflow

1. Run `npm run setup` on first use.
2. Run `npm run visual:ui`.
3. Open `http://127.0.0.1:4317/`.
4. Enter baseline and target URLs.
5. Choose the URL source.
6. Run `Extract URLs`, `Compare`, or `Extract + Compare`.
7. Use live filters to focus on failed, successful, error, or skipped rows while the run continues.
8. Open the generated report for lightbox review and final triage.

### Manual Relative URL Workflow

Manual lists are supported in both the dashboard and CLI. This is useful when reviewers already have a fixed route checklist and want to run the same paths across multiple domains.

Accepted entries include:

- `/`
- `/about`
- `contact`
- full URLs copied from another domain

The tool normalizes each entry to a relative path and rebuilds matching baseline and target URLs for the current run.

CLI example:

```sh
npm run visual:all -- --baseline=https://example.com --target=https://preview.example.com --url-source=manual --manual-urls=/,/about,/contact
```

### CLI Workflow

```sh
npm run setup
npm run visual:urls -- --baseline=https://example.com --target=https://preview.example.com --url-source=sitemap --max-pages=100
npm run visual:test -- --baseline=https://example.com --target=https://preview.example.com
npm run visual:report
```

Full workflow:

```sh
npm run visual:all -- --baseline=https://example.com --target=https://preview.example.com
```

## Outputs

Run outputs are written to:

```text
visual-test-results/YYYY-MM-DD/run-###/
```

The latest completed run is also available at:

```text
visual-test-results/latest/
```

Generated run artifacts include:

- `index.html` reviewer report.
- `summary.md` Markdown summary.
- `run.json` run metadata.
- `metadata.json` per comparison.
- baseline screenshots.
- target screenshots.
- diff screenshots.

URL extraction artifacts include:

- `visual/urls.json`
- `visual/urls.md`
- `visual/pages.json`
- `visual/pages.md`

Generated output folders are ignored by git.

## Compatibility

Supported runtime:

- Node.js 18 or newer.

Validated compatibility matrix:

- Node 18, 20, and 22.
- Linux, Windows, and macOS through GitHub Actions.

Command compatibility notes:

- npm scripts use `tsx` directly for compatibility across supported Node versions.
- macOS and Linux users can set `PORT=4320 npm run visual:ui`.
- Windows PowerShell users can set `$env:PORT=4320; npm run visual:ui`.
- Windows Command Prompt users can set `set PORT=4320 && npm run visual:ui`.

## Configuration

Common configuration can be controlled through `visual.config.ts`, CLI flags, or environment variables:

- baseline URL
- target URL
- URL source
- sitemap paths
- manual relative URL list
- max pages
- visual threshold
- pixel threshold
- retry count
- viewport list
- screenshot wait strategy
- image readiness waiting
- cookie banner selectors
- mask selectors
- hide selectors

## Privacy And Repo Hygiene

- Tracked source files do not require client-specific URLs.
- Saved dashboard presets are written to `visual/presets.json`, which is ignored by git.
- Generated reports and screenshots are ignored by git.
- Defaults use generic `example.com` values.
- Release notes avoid project-specific comparison domains.

## Upgrade Notes

- Existing `v0.1.0` reports can be regenerated with `npm run visual:report -- --run-dir <run-dir>` to receive the new report UI.
- Older runs cannot recover true historical durations or final post-redirect URLs if those fields were not captured at run time.
- New runs capture duration and final URL metadata automatically.
- No config migration is required from `v0.1.0`.

## Known Notes

- The dashboard is intended for local trusted use and binds to `127.0.0.1` by default.
- Very dynamic pages may still need masks, hidden selectors, retries, or threshold tuning.
- Final HTML and Markdown reports are written after the comparison job completes, while the live dashboard updates during the run.
